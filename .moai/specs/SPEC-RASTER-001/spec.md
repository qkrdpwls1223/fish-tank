# SPEC-RASTER-001: 물고기 그림 벡터 획 → 래스터(비트맵) 기반 개편

## HISTORY

- 2026-07-10 (v0.1.0): 최초 작성. 세 개의 사용자 요구(30마리+ 어항 렉 수정 / 페인트통 플러드필 / 이미지 업로드 물고기)를 하나의 아키텍처 변경(그림 표현을 벡터 획 → 래스터 비트맵으로 전환)으로 수렴시켜 EARS 형식으로 정리. 기존 벡터 물고기(SPEC-TANK-001·SPEC-CATCH-001·내 어항)의 **비회귀**를 최상위 제약으로 명시. 저장 포맷·하위호환·꼬리/입 애니메이션 보존·성능·보안 검증을 각각 검증 가능한 요구사항/NFR로 포함. O(n²) 무리짓기(flocking) 최적화는 본 SPEC 범위 밖 "고려사항"으로만 표시.

---

## Goal (목표)

물고기 **그림 표현 모델**을 현재의 **벡터 획(strokes) 기반**에서 **래스터(픽셀 비트맵) 기반**으로 전환한다. 이 단일 아키텍처 변경으로 서로 얽힌 세 가지 목표를 동시에 해결한다.

1. **성능(렉) 수정**: 어항 물고기가 30마리 수준부터 발생하는 프레임 저하를, 매 프레임 모든 물고기의 벡터 획을 재계산·재-path 렌더링하는 대신 **한 번 렌더한 비트맵을 캐싱 후 프레임마다 blit(복사)** 하는 방식으로 제거한다.
2. **페인트통(플러드필)**: 사용자가 원하는 "닫힌 영역 채우기"는 래스터 캔버스에서만 가능하므로, 래스터 전환을 전제로 페인트통 도구를 제공한다.
3. **이미지 업로드 물고기**: 그리기 대신 이미지 파일을 업로드해 물고기로 등록하는 경로를 제공한다. 업로드 이미지는 래스터 비트맵과 **동일한 저장 모델**로 자연스럽게 통합된다.

정서적/제품 핵심(Product Core): "물고기를 그리는 방법이 더 자유로워지고(채우기·이미지 업로드), 어항은 물고기가 많아져도 부드럽게 헤엄친다 — 그러면서 이미 헤엄치던 물고기는 하나도 사라지거나 어색해지지 않는다."

## Target Audience (대상 사용자)

- 공유 어항(SPEC-TANK-001)·낚시/수집함(SPEC-CATCH-001)·내 어항을 이미 사용하는 사내 임직원 (동일 인증 컨텍스트)
- 사용 맥락: 물고기를 더 다양한 방식(채우기, 사진/그림 업로드)으로 만들고 싶은 사용자, 물고기 수가 많아진 어항을 쾌적하게 감상하려는 사용자

## Scope (범위)

### In Scope (포함)

- 그림 표현 모델을 래스터(비트맵) 기반으로 확장하고, **신규 물고기는 래스터로 저장**
- 그리기 캔버스에 **페인트통(플러드필)** 도구 추가
- **이미지 파일 업로드**로 물고기를 등록하는 경로 추가 (공유 어항 및 내 어항 모두)
- 어항/수집함/스냅샷 렌더러의 **오프스크린 비트맵 캐싱 후 blit** 도입 (성능)
- 기존 **벡터 물고기(strokes 포맷)의 계속 렌더링(비회귀)** 및 포맷 공존
- 래스터/업로드 페이로드에 대한 **서버 측 독립 보안 검증**(형식/크기/디코드 안전성) — 기존 NFR-SEC-003 유지·강화
- 래스터에서도 **꼬리 파닥임 / 입 벌림 애니메이션**을 보존 (어항 생동감 비회귀)
- (필요 시) DB 스키마 마이그레이션 `005` 추가

### Out of Scope (제외 — 상세는 아래 Exclusions 참조)

- 무리짓기(flocking) O(n²) 알고리즘 최적화, 기존 벡터 물고기의 강제 래스터 마이그레이션, 그림 편집(다시 그리기), 애니메이션 GIF/동영상 업로드 등

## Functional Requirements (EARS)

> EARS 패턴 표기: **[Ubiquitous]** 항시, **[Event]** 이벤트 기반(WHEN), **[State]** 상태 기반(WHILE), **[Unwanted]** 비정상 처리(IF...THEN), **[Optional]** 선택 기능(WHERE)

### 래스터 저장 포맷 (RAS)

- **REQ-RAS-001 [Ubiquitous]**: 시스템은 물고기 그림을 **래스터(픽셀 비트맵)** 로 표현·저장할 수 있어야 한다. 래스터 표현은 **자기완결적(self-contained)** 이어서 외부 리소스 참조 없이 단독으로 디코드·렌더 가능해야 한다.
- **REQ-RAS-002 [Ubiquitous]**: 시스템은 래스터 물고기를 기존 벡터 포맷과 **구분 가능한 버전/타입 식별자**(예: `version` 필드)로 저장해야 하며, 어떤 표현인지 데이터만 보고 판별할 수 있어야 한다. (현재 검증기는 `version === 1`만 통과시키므로 신규 포맷은 별도 버전으로 분기한다.)
- **REQ-RAS-003 [Event]**: 사용자가 래스터 물고기를 등록할 때(WHEN), 시스템은 정해진 **캔버스 해상도 상한** 이내로 정규화하여 저장해야 한다.
- **REQ-RAS-004 [Unwanted]**: 래스터 페이로드가 형식 위반이거나 크기 상한을 초과하거나 디코드 불가능하면(IF), 시스템은(THEN) 등록을 거부하고 사유를 안내하며, 저장·렌더를 수행하지 않아야 한다.
- **REQ-RAS-005 [Ubiquitous]**: 시스템은 래스터 물고기의 **투명 배경**을 지원해야 한다(어항 배경 위에 물고기 형상만 자연스럽게 보이도록).

### 하위호환 및 마이그레이션 (COMPAT)

- **REQ-COMPAT-001 [Ubiquitous] — 포맷 공존 불변식**: 시스템은 **기존 벡터(strokes) 물고기와 신규 래스터 물고기를 동시에 렌더**할 수 있어야 한다. 저장된 표현의 버전/타입 식별자에 따라 렌더 경로를 분기한다.
- **REQ-COMPAT-002 [Ubiquitous] — 벡터 비회귀 불변식**: 이미 저장된 벡터 물고기는 본 개편 이후에도 **모양·꼬리 파닥임·입 벌림·삭제/먹이주기 동작이 변하지 않아야** 한다. 어떤 기존 물고기도 사라지거나 깨져서 표시되어서는 안 된다.
- **REQ-COMPAT-003 [Ubiquitous]**: 신규로 그리거나 업로드하는 물고기는 **래스터 포맷으로 저장**해야 한다(벡터 신규 생성 경로는 래스터로 대체).
- **REQ-COMPAT-004 [Optional]**: 데이터 정합성을 위해 스키마 변경이 필요한 경우(WHERE), 시스템은 신규 마이그레이션(`005_*`)으로 이를 반영해야 한다. 기존 벡터 물고기 행을 **강제로 래스터로 변환하지 않는다**(마이그레이션은 스키마 확장에 한정, 데이터 보존).

### 꼬리/입 애니메이션 보존 (ANIM)

- **REQ-ANIM-001 [Ubiquitous]**: 시스템은 래스터 물고기에 대해서도 **꼬리 경계(tailFraction)** 와 **입 경계(mouthFraction)** 개념을 유지해야 한다. 두 경계는 비트맵을 세로선 기준으로 분할하는 데 쓰인다.
- **REQ-ANIM-002 [State]**: 어항에서 물고기가 헤엄치는 동안(WHILE), 시스템은 래스터 비트맵의 **꼬리 영역(꼬리 경계선 왼쪽)** 을 시간 기반으로 흔들어(shear/휨) 파닥임을 표현해야 한다.
- **REQ-ANIM-003 [Event]**: 물고기가 먹이에 반응할 때(WHEN), 시스템은 래스터 비트맵의 **입 영역(입 경계선 오른쪽)** 을 벌렸다 다무는 애니메이션으로 표현해야 한다.
- **REQ-ANIM-004 [Event]**: 사용자가 래스터/업로드 물고기를 등록할 때(WHEN), 시스템은 벡터 그리기와 동일하게 **꼬리/입 경계 위치를 지정**할 수 있게 제공해야 하며, 지정하지 않은 경우 기존 기본값(꼬리 0.4 / 입 0.72)으로 폴백해야 한다.

### 페인트통 / 플러드필 (FILL)

- **REQ-FILL-001 [Event]**: 사용자가 페인트통 도구로 캔버스의 한 지점을 채울 때(WHEN), 시스템은 그 지점과 색이 연속으로 이어진 **닫힌 영역(연결 성분)** 을 선택 색으로 채워야 한다(래스터 flood fill).
- **REQ-FILL-002 [Ubiquitous]**: 시스템은 안티에일리어싱 경계에서 자연스럽게 동작하도록 **색 허용 오차(tolerance)** 를 적용해 채움 영역을 판정해야 한다.
- **REQ-FILL-003 [Unwanted]**: 채우기 시작 지점이 이미 목표 색과 동일하거나 채울 영역이 없으면(IF), 시스템은(THEN) 캔버스를 변경하지 않고 무동작(no-op)이어야 한다(무한 루프/오작동 방지).
- **REQ-FILL-004 [State]**: 그리기 캔버스가 활성인 동안(WHILE), 시스템은 페인트통 동작에 대해 기존과 동일하게 **실행 취소(undo)** 를 제공해야 한다.

### 이미지 업로드 물고기 (UPLOAD)

- **REQ-UPLOAD-001 [Event]**: 사용자가 이미지 파일을 업로드해 물고기를 등록할 때(WHEN), 시스템은 그 이미지를 물고기 래스터 비트맵으로 사용해야 한다.
- **REQ-UPLOAD-002 [Ubiquitous]**: 시스템은 **허용된 이미지 포맷만** 업로드를 받아들여야 한다(권장: PNG/JPEG/WebP 정지 이미지). 그 외 포맷은 거부한다.
- **REQ-UPLOAD-003 [Ubiquitous]**: 시스템은 업로드 이미지를 **캔버스 규격(해상도 상한)으로 리사이즈/정규화**하여 저장해야 한다.
- **REQ-UPLOAD-004 [Event]**: 이미지를 업로드할 때(WHEN), 시스템은 사용자가 **꼬리/입 경계 위치를 지정**할 수 있게 제공해야 한다(REQ-ANIM-004와 동일 규칙, 미지정 시 기본값 폴백).
- **REQ-UPLOAD-005 [Ubiquitous]**: 시스템은 이미지 업로드 물고기 등록을 **공유 어항과 내 어항 모두**에서 지원해야 한다. 각 스코프의 기존 저장/공개 규칙(공유=브로드캐스트, 내 어항=비공개)은 그대로 유지된다.
- **REQ-UPLOAD-006 [Unwanted]**: 업로드 파일이 허용 포맷이 아니거나 크기 상한을 초과하거나 유효한 이미지로 디코드되지 않으면(IF), 시스템은(THEN) 등록을 거부하고 사유를 안내해야 한다.

### 렌더 파이프라인 캐싱 (RENDER)

- **REQ-RENDER-001 [Ubiquitous]**: 시스템은 각 물고기 비트맵을 **오프스크린 캔버스/이미지에 한 번 렌더한 뒤 캐싱**하고, 프레임마다 재렌더 대신 **캐시된 비트맵을 blit** 해야 한다. (꼬리/입 애니메이션 변형은 blit 시점의 변환/부분 재합성으로 적용한다.)
- **REQ-RENDER-002 [Event]**: 물고기의 시각적 표현이 바뀌는 경우(예: 새 물고기 추가, 삭제)에만(WHEN), 시스템은 해당 캐시를 생성/무효화해야 한다(불필요한 재렌더 방지).
- **REQ-RENDER-003 [Ubiquitous]**: 오프스크린 캐싱은 **어항(FishTank) · 내 어항(MyTank) · 스냅샷(renderSnapshot: 수집함/낚시)** 세 렌더 경로 모두에 일관되게 적용되어야 한다.

## Non-Functional Requirements (NFR)

### 렌더링 성능 (Performance)

- **NFR-PERF-001 [State]**: 어항에 물고기가 다수 존재하는 동안(WHILE), 시스템은 프레임 저하 없이 렌더해야 한다. **목표: 물고기 50마리에서 뚜렷한 렉 없음, 100마리에서도 사용 가능한 프레임레이트 유지**(기존 벡터 30마리 수준 저하 대비 명백한 개선). 동시 애니메이션 상한(`MAX_ANIMATED`) 정책은 유지·조정 가능.
- **NFR-PERF-002 [Ubiquitous]**: 오프스크린 캐싱은 물고기 1마리당 상수 시간 blit에 수렴해야 하며, 매 프레임 비용이 획 수·점 수에 비례해서는 안 된다(벡터 재-path 비용 제거).

### 보안 / 검증 (Security — NFR-SEC-003 유지·강화)

- **NFR-SEC-001 [Ubiquitous]**: 서버가 최종 권한을 가진다. 시스템은 래스터/업로드 페이로드에 대해 **서버 측에서 독립적으로 형식·크기·디코드 안전성**을 검증해야 하며, 클라이언트가 보낸 값을 신뢰해서는 안 된다.
- **NFR-SEC-002 [Unwanted]**: 래스터 페이로드가 data URL/base64 형태인 경우(IF), 시스템은(THEN) **MIME 타입·매직바이트(파일 시그니처)를 확인**하고 허용 포맷만 통과시켜야 한다. 스크립트/URL/실행 가능 콘텐츠가 이미지로 위장해 주입되는 것을 차단해야 한다(기존 hex color 화이트리스트가 담당하던 주입 방어를 래스터/업로드 경로에서도 유지).
- **NFR-SEC-003 [Ubiquitous]**: 시스템은 업로드/래스터 이미지를 **클라이언트 캔버스에서 규격 리사이즈 후 PNG data URL로 정규화**하여 전송하고, **서버는 형식(매직바이트/data URL 접두)·크기·픽셀 해상도 상한을 독립 검증**하여 통과/거부만 판단한다. 서버측 네이티브 재인코드 라이브러리(sharp/jimp)는 도입하지 않는다(결정 2026-07-10). 캔버스 재그림 과정에서 원본 이미지가 사실상 PNG로 재구성되어 대부분의 임베디드 메타데이터/악성 페이로드가 제거되나, 서버가 바이트 단위 재인코드로 완전 제거하지는 않음을 명시한다(잔여 위험은 매직바이트 검증·크기 상한·hex 외 콘텐츠 차단으로 완화).
- **NFR-SEC-004 [Ubiquitous]**: 시스템은 래스터/업로드 그림에 **직렬화 크기 상한 및 픽셀 해상도 상한**을 적용해야 한다(DoS/저장소 남용 방지). 상한값은 벡터 대비 재산정한다(아래 NFR-STORAGE 참조).

### 저장/용량 (Storage)

- **NFR-STORAGE-001 [Ubiquitous]**: 시스템은 래스터 그림의 저장 크기 상한을 정의해야 한다. 벡터 포맷의 `maxBytes = 100KB`는 base64 PNG에는 부족할 수 있으므로, **래스터 전용 상한을 별도로 상향 정의**하되 무제한이어서는 안 된다. 해상도 상한은 기존 `maxCanvas = 2000px`와 정합하게 유지한다.

### 비회귀 / 공존 (Non-Regression & Coexistence)

- **NFR-REG-001 [Ubiquitous]**: 본 개편은 SPEC-TANK-001(공유 어항: 추가/삭제/먹이주기/실시간 전파, `toPublicFish`/`toViewerFish` 투영, 소유권 삭제 규칙), SPEC-CATCH-001(낚시/수집함: 비파괴·프라이버시·스냅샷 생존), 내 어항(비공개 스코프·장식·scale·드래그) 동작을 **변경·저하시키지 않아야** 한다.
- **NFR-REG-002 [Ubiquitous]**: 기존 테스트 스위트(server/client)는 **개편 후에도 전부 통과**해야 한다. 신규 기능은 기존 REQ/NFR에 대해 비회귀여야 한다.

### 접근성 (Accessibility)

- **NFR-A11Y-001 [Ubiquitous]**: 페인트통 도구와 이미지 업로드 경로에 대해 접근성 대체 수단을 제공해야 한다(키보드 조작 경로, 명확한 라벨/안내 문구, 충분한 색 대비 — SPEC-TANK-001 NFR-A11Y-001 준용).

## Exclusions (What NOT to Build)

초기 범위에서 명시적으로 제외한다. (범위 크립 방지)

- **무리짓기(flocking) 알고리즘 최적화**: `client/src/tank/tankModel.js`의 이웃 계산이 O(n²)라 100마리+에서 별도 병목이 될 수 있으나, 본 SPEC은 **렌더 파이프라인(캐싱/blit)** 개선에 집중한다. flocking 최적화(공간 분할/그리드 등)는 **별도 SPEC**으로 다룬다(고려사항 섹션 참조).
- **기존 벡터 물고기의 강제 래스터 변환**: 이미 저장된 벡터 물고기는 그대로 벡터로 렌더한다. 일괄 데이터 마이그레이션(벡터→래스터)은 하지 않는다 (REQ-COMPAT-002/004).
- **그림 편집/다시 그리기**: 등록된 물고기(벡터·래스터·업로드 무관)의 재편집은 제공하지 않는다(기존 REQ-OWN-005 유지 — 삭제만 지원).
- **애니메이션 이미지/동영상 업로드**: GIF 애니메이션, 동영상, SVG 등 실행 가능/애니메이션 포맷 업로드는 만들지 않는다(정지 래스터 이미지만).
- **서버측 이미지 자동 보정/필터/배경 제거 AI**: 업로드 이미지에 대한 자동 누끼/스타일 변환 등은 초기 범위 밖이다.
- **레이어/브러시 고급 편집 도구**: 페인트통 외의 그라디언트·스탬프·다중 레이어 등 고급 페인팅 기능은 만들지 않는다.

## Considerations (고려사항 — 본 SPEC 범위 밖, 별도 SPEC 여지)

- **무리짓기 O(n²)**: `tankModel.js`의 이웃 기반 정렬/응집 계산이 물고기 수의 제곱에 비례한다. 렌더 캐싱으로 렌더 병목은 해소되어도 100마리+에서는 시뮬레이션 비용이 남을 수 있다. 공간 분할(grid/quadtree)·이웃 상한·틱 분할 등은 **후속 SPEC(예: SPEC-FLOCK-00X)** 로 분리 권장.
- **저장소 용량 증가**: 래스터(특히 업로드 이미지)는 벡터보다 저장 크기가 크다. 장기적으로 보존/정리 정책(SPEC-TANK-001 REQ-PERSIST-003)과 연동 검토 여지.

## Assumptions (가정)

- 래스터/업로드 경로는 SPEC-TANK-001·SPEC-CATCH-001·내 어항과 동일한 인증 컨텍스트(`authRequired` → `req.user.userId`)를 재사용한다.
- 저장은 기존과 동일한 PostgreSQL(`pg` Pool)을 사용한다. 그림 데이터는 기존과 동일하게 물고기 레코드의 `drawing`(JSONB) 필드에 저장하되, 래스터 표현을 담을 수 있게 포맷을 확장한다.
- 클라이언트 사전 검증과 서버 검증은 동일 규칙을 공유하되(현 구조 유지), **서버가 최종 권한**이다.
- 래스터 표현의 **권장 구현**은 자기완결적 PNG data URL(base64)이며, 신규 `version` 값(예: 2)으로 벡터와 분기한다. 최종 포맷·상한·해상도 규격은 Run 단계에서 확정한다.
- 테스트는 기존과 동일하게 `node node_modules/vitest/dist/cli.js run --root server` / `--root client` 로 실행한다.

## Implementation Plan (계획)

### 순서 제약 (필수)

**백엔드(검증/저장/마이그레이션) → 프론트(그리기 UI·모델) → 렌더러 3곳 → 통합 검증** 순서로 진행한다. 보안 경계(서버 검증)와 저장 포맷을 먼저 확정해야 프론트가 신뢰 가능한 계약 위에서 구현되기 때문이다.

### 마일스톤 (우선순위 순)

- **M1 (Priority High) — 백엔드 저장 포맷·검증·마이그레이션**: 래스터 버전 식별자 확정, `validateDrawing.js`에 래스터 검증 분기 추가(형식/크기/MIME·매직바이트/디코드), 필요 시 `005_*` 마이그레이션, 업로드 수신·서버측 재인코드/리사이즈. 벡터 검증 비회귀 확인. (REQ-RAS-*, REQ-COMPAT-004, NFR-SEC-*, NFR-STORAGE-001)
- **M2 (Priority High) — 프론트 그림 모델·그리기 UI**: `drawingModel.js` 래스터 상태·상한·페인트통 액션, `DrawingCanvas.jsx` 페인트통·이미지 업로드·꼬리/입 경계 지정 UI, 클라 사전 검증 정합화. (REQ-FILL-*, REQ-UPLOAD-*, REQ-ANIM-004, REQ-COMPAT-003)
- **M3 (Priority High) — 렌더러 3곳 오프스크린 캐싱 + 포맷 분기**: `FishTank.jsx`·`MyTank.jsx`·`renderSnapshot.js`에 캐싱 blit 도입, 벡터/래스터 분기, 래스터 꼬리/입 애니메이션. (REQ-RENDER-*, REQ-COMPAT-001/002, REQ-ANIM-001/002/003, NFR-PERF-*)
- **M4 (Priority Medium) — 통합 검증**: 벡터 비회귀(기존 물고기 정상 표시), 신규 래스터 저장·렌더, 페인트통, 이미지 업로드, 성능 목표(50/100마리), 보안 검증을 acceptance 기준으로 확인. 기존 server/client 테스트 전부 통과. (NFR-REG-*)

### 기술적 접근 요지 (What/Why 수준 — How는 Run에서 확정)

- 포맷 공존: 저장 데이터의 `version`으로 렌더/검증 경로를 분기(벡터=1, 래스터=신규 값).
- 캐싱: 물고기당 오프스크린 캔버스 1개를 생성/무효화하고, 프레임 루프는 blit + 꼬리/입 변형만 수행.
- 애니메이션 보존: `tailFraction`/`mouthFraction` 경계로 비트맵을 세로 분할해 꼬리 shear·입 gape를 적용.

### 위험 (Risks)

- **렌더러 3곳 동시 변경**에 따른 회귀 위험(가장 큼) → M3에서 포맷 분기·캐싱을 공통화하고 벡터 경로를 우선 보존.
- **보안 표면 확대**(임의 이미지 수신) → M1에서 서버 재인코드/정규화·매직바이트 검증을 계약으로 고정.
- **저장 크기 증가** → NFR-STORAGE-001 상한을 명시적으로 정의.

## Acceptance Criteria (EARS, 검증 가능)

> 검증 방법 표기: 자동 테스트는 `node node_modules/vitest/dist/cli.js run --root server|client` 로 실행. 성능은 계측(프레임레이트/프레임 시간) 기준.

### 벡터 비회귀

- **AC-001 [Ubiquitous]**: 개편 이후에도 **기존 벡터 물고기가 동일하게 렌더**되어야 한다. 검증: 기존 벡터 그림 fixture가 어항/내 어항/스냅샷에서 모양·꼬리 파닥임·입 벌림이 이전과 동일하게 표시되고, 기존 server/client 테스트가 **전부 통과**한다(NFR-REG-002).

### 신규 래스터 저장·렌더

- **AC-002 [Event]**: 사용자가 래스터 물고기를 등록하면(WHEN), 시스템은 이를 래스터 포맷(신규 version)으로 저장하고 어항에서 헤엄치게 해야 한다. 검증: 등록 후 저장 데이터의 포맷 식별자가 래스터이고, 렌더 경로가 래스터 분기를 타며 정상 표시된다.
- **AC-003 [Unwanted]**: 유효하지 않은 래스터 페이로드(형식/크기 초과/디코드 실패)를 저장하려 하면(IF), 서버 검증이 이를 **거부**해야 한다. 검증: `validateDrawing` 래스터 분기 단위 테스트가 정상/비정상 케이스를 구분한다.
- **AC-004 [Event]**: 래스터 물고기가 어항에서 헤엄칠 때(WHEN), **꼬리 파닥임·입 벌림 애니메이션이 표시**되어야 한다. 검증: tailFraction/mouthFraction 경계로 꼬리/입 영역이 각각 변형되는 것을 렌더 테스트/계측으로 확인.

### 페인트통 (플러드필)

- **AC-005 [Event]**: 사용자가 닫힌 영역 안을 페인트통으로 클릭하면(WHEN), 그 연결 영역이 선택 색으로 채워져야 한다. 검증: flood fill 순수 함수 단위 테스트(연결 성분/경계/tolerance).
- **AC-006 [Unwanted]**: 이미 목표 색인 지점을 채우려 하면(IF), 캔버스가 변하지 않아야(no-op) 한다. 검증: 동일 색 시작점 테스트가 무변경을 확인하고 undo로 되돌릴 수 있다(REQ-FILL-004).

### 이미지 업로드

- **AC-007 [Event]**: 사용자가 허용 포맷 이미지를 업로드하면(WHEN), 캔버스 규격으로 정규화되어 물고기로 등록되어야 하며 **공유 어항·내 어항 모두**에서 동작해야 한다. 검증: 업로드→리사이즈→저장 경로 테스트, 두 스코프 각각 등록 확인.
- **AC-008 [Unwanted]**: 허용되지 않은 포맷/크기 초과/디코드 불가 파일을 업로드하면(IF), 시스템은 이를 **거부**하고 사유를 안내해야 한다. 검증: 비허용 MIME/매직바이트/초과 크기 케이스 거부 테스트.

### 성능

- **AC-009 [State]**: 물고기 **50마리에서 뚜렷한 렉이 없고, 100마리에서도 사용 가능한 프레임레이트**를 유지해야 한다. 검증: 프레임 시간 계측이 벡터 기준(30마리 저하) 대비 명확히 개선되고, 프레임당 비용이 획/점 수에 비례하지 않음을 확인(NFR-PERF-002).

### 보안 검증

- **AC-010 [Unwanted]**: data URL/base64 래스터가 실제로는 이미지가 아니거나 스크립트/실행 콘텐츠를 위장하면(IF), 서버가 **data URL 접두·MIME·매직바이트 검증 및 크기/해상도 상한**으로 이를 차단해야 한다. 검증: 위장 페이로드(잘못된 매직바이트/비허용 MIME/초과 크기) 거부 + 정상 PNG data URL 통과 단위 테스트(NFR-SEC-002/003).

## Suggested Implementation Surface (제안 — 최종 형태는 Run 단계에서 확정)

> 이 절은 구현 힌트이며 계약이 아니다. 구체 스키마/함수/API 형식은 Run 단계에서 결정한다.

- **래스터 포맷(권장)**: `{ version: 2, kind: "raster", width, height, tailFraction, mouthFraction, image: "<PNG data URL>" }`. 벡터(`version: 1`)와 `version`으로 분기.
- **검증 확장(권장)**: `validateDrawing`에 `version === 2` 분기 — 해상도 상한(≤ 2000px), 래스터 전용 크기 상한(NFR-STORAGE-001), data URL 접두 검사 + base64 디코드 + 매직바이트(PNG/JPEG/WebP) 확인.
- **정규화(결정됨)**: 클라이언트 캔버스에서 규격 리사이즈 후 `canvas.toDataURL("image/png")`로 PNG data URL 생성 → 전송. 서버는 재인코드 없이 검증만(매직바이트/크기/해상도). sharp/jimp 미도입.
- **렌더 캐싱(권장)**: 물고기 id별 오프스크린 캔버스 캐시 맵. 벡터는 캐시 캔버스에 한 번 stroke, 래스터/업로드는 이미지 디코드 후 캐시에 그림. 프레임 루프는 캐시 blit + 꼬리 shear/입 gape 변형만.
- **페인트통(권장)**: 스캔라인 flood fill 순수 함수(픽셀 버퍼 + tolerance), `drawingModel.js` 액션으로 편입해 undo 스택과 통합.
- **마이그레이션(필요 시)**: `server/migrations/005_*.sql` — `drawing` JSONB는 스키마 무관하므로 컬럼 추가가 불필요할 수 있음. 필요한 인덱스/제약만 반영, 기존 데이터 보존.
