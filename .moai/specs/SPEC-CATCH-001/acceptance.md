# 인수 기준: SPEC-CATCH-001 물고기 낚시 & 개인 수집함

> Given-When-Then 형식. 각 시나리오는 관찰 가능한 증거를 기준으로 검증한다.

## Scenario 1: 타인 물고기 낚기 — 비파괴 수집 (REQ-CATCH-001, REQ-CATCH-002, REQ-CATCH-003)

- **Given** 사용자 B가 그린 물고기가 공유 어항에 있고, 사용자 A가 로그인해 어항을 보고 있다
- **When** A가 B의 물고기를 낚는다
- **Then** A의 수집함에 그 물고기의 복사본이 추가되고, 공유 어항의 원본 물고기는 삭제·변경 없이 그대로 남아 계속 헤엄친다(낚기 전/후 `GET /api/fish` 결과가 동일)

## Scenario 2: 본인 물고기 낚기 (REQ-CATCH-002)

- **Given** 사용자 A가 직접 그려 어항에 있는 본인 물고기가 있다
- **When** A가 자신의 물고기를 낚는다
- **Then** A의 수집함에 복사본이 추가되고, 원본은 어항에 그대로 유지된다(본인 물고기도 낚기 허용, 비파괴)

## Scenario 3: 완전 비공개 — 브로드캐스트 없음 (REQ-PRIV-001, REQ-PRIV-002)

- **Given** 사용자 A와 사용자 B가 동시에 공유 어항을 보고 있다
- **When** A가 어떤 물고기를 낚는다
- **Then** B의 어항 뷰에는 아무 변화도 발생하지 않고(실시간 이벤트 미수신), 낚은 사실과 A의 수집함은 A에게만 보인다

## Scenario 4: 수집함 조회 — 본인 스코프 & 스냅샷 렌더링 (REQ-COLL-001, REQ-COLL-002, REQ-COLL-003, REQ-COLL-004, REQ-PRIV-003)

- **Given** 사용자 A가 물고기 여러 마리를 낚아둔 상태다
- **When** A가 공유 어항과 분리된 "내 수집함" 화면을 연다
- **Then** A가 낚은 물고기들이 낚은 시점의 스냅샷으로 렌더링되고 각 항목에 낚은 시각(caught_at)이 표시되며, 목록은 A 본인이 낚은 것만 포함한다(타인 수집 미포함)

## Scenario 5: 스냅샷-삭제-생존 — 원본 삭제 후에도 간직 (REQ-SNAP-001, REQ-SNAP-002, REQ-SNAP-003)

- **Given** 사용자 A가 사용자 B의 물고기를 낚아 수집함에 가지고 있다
- **When** 원본 소유자 B가 공유 어항에서 자신의 그 물고기를 삭제한다
- **Then** 공유 어항에서는 원본이 사라지지만, A의 수집함에 있는 복사본은 영구히 그대로 남아 계속 조회·렌더링된다

## Scenario 6: 소유자 신원 비노출 (REQ-PRIV-004, SPEC-TANK-001 REQ-OWN-004)

- **Given** 익명으로 등록된 물고기와 이름 표시 물고기가 어항에 있다
- **When** 사용자 A가 각각을 낚아 수집함에서 확인한다
- **Then** 낚시·수집함 응답 어디에도 원본의 내부 `owner_id`가 노출되지 않고, 익명 물고기는 여전히 "익명"으로 표시된다(누가 그렸는지 드러나지 않음)

## Scenario 7: 존재하지 않는 물고기 낚기 거부 (REQ-CATCH-004)

- **Given** 사용자 A가 이미 삭제되었거나 존재하지 않는 물고기 ID로 낚시를 시도한다
- **When** 낚시 요청이 서버에 도달한다
- **Then** 시스템은 낚시를 거부하고 "더 이상 존재하지 않음"을 안내하며, A의 수집함에는 항목이 생성되지 않는다

## Scenario 8: 중복 낚기 멱등 처리 (REQ-CATCH-005)

- **Given** 사용자 A의 수집함에 특정 원본 물고기의 복사본이 이미 있다
- **When** A가 같은 원본 물고기를 다시 낚는다
- **Then** 수집함에 중복 항목이 생기지 않고(멱등), "이미 수집함"임이 안내된다

## Edge Cases (엣지 케이스)

- **비파괴 불변식**: 낚기는 원본 `fish` 레코드에 UPDATE/DELETE를 발생시키지 않는다. 낚기 전/후 공유 어항 스냅샷이 동일함을 테스트로 증명한다. (REQ-CATCH-003)
- **스냅샷 독립성**: 원본이 삭제되어도 수집 복사본은 생존한다. `caught_fish`는 원본 삭제에 cascade되지 않는다. (REQ-SNAP-002, REQ-SNAP-003)
- **낚는 순간 원본 삭제 경합**: `findById`가 원본을 찾지 못하면 낚시를 거부하고 항목을 만들지 않는다. 이미 삽입된 스냅샷은 이후 원본 삭제와 무관하게 유지된다. (REQ-CATCH-004)
- **중복 낚기**: (catcher_id, source_fish_id) 유일성으로 동일 물고기 재낚기를 멱등 처리한다. (REQ-CATCH-005)
- **프라이버시 스코프**: 사용자는 타인의 수집함을 조회할 수 없다. 위조 신원으로 타인 수집함 접근을 시도해도 서버가 인증 신원으로 스코프해 차단한다. (REQ-PRIV-003, NFR-SEC-003)
- **브로드캐스트 미발생**: 낚시 동작이 어떤 실시간 이벤트도 발생시키지 않으며, 다른 접속 사용자의 어항 상태에 영향을 주지 않는다. (REQ-PRIV-002)
- **owner_id 비노출**: 낚시/수집함의 모든 응답에서 내부 `owner_id`가 제거된다(공개 투영 원칙 준용). (REQ-PRIV-004)
- **클라이언트 콘텐츠 주입 방지**: 낚시 요청은 원본 물고기 ID만 사용하고, 스냅샷은 서버 저장 데이터에서만 유도한다(클라이언트가 임의 그림을 수집함에 넣을 수 없음). (NFR-SEC-002)
- **빈 수집함**: 낚은 물고기가 없으면 빈 상태 안내를 표시한다. (REQ-COLL-005)
- **기존 어항 비회귀**: 낚시 도입 후에도 공유 어항의 추가/삭제/먹이/실시간/투영 동작이 변하지 않는다. (NFR-COMPAT-001)

## Quality Gate Criteria (품질 게이트)

- 모든 EARS 기능 요구사항(REQ-*)에 대응하는 인수 시나리오 또는 엣지 케이스가 존재한다.
- 테스트 커버리지 목표: 85% 이상 (`quality.yaml` 기준). 서버는 supertest, 클라이언트는 Vitest 사용.
- LSP 오류/타입 오류/린트 오류 0 (Run 단계 기준).
- **비파괴 불변식 증명**: 낚기 전/후 공유 어항 상태가 동일하고 원본 레코드가 무변경임을 테스트로 보인다(REQ-CATCH-003).
- **스냅샷 생존 증명**: 원본 삭제 후 수집 복사본이 여전히 조회됨을 테스트로 보인다(REQ-SNAP-002).
- **프라이버시 증명**: 낚시가 브로드캐스터를 호출하지 않고(REQ-PRIV-002), 수집함 조회가 본인 스코프로 제한되며(REQ-PRIV-003), 응답에 owner_id가 없음(REQ-PRIV-004)을 테스트로 보인다.
- **비회귀 증명**: SPEC-TANK-001의 기존 테스트(client/server)가 낚시 도입 후에도 모두 통과한다(NFR-COMPAT-001).

## Definition of Done (완료 정의)

- [ ] 공유 어항의 어떤 물고기든(본인/타인) 낚아 수집함에 복사본 추가
- [ ] 낚아도 원본 물고기가 어항에 그대로 유지됨(비파괴 불변식 검증)
- [ ] 낚은 시점 스냅샷을 독립 저장하고 "내 수집함"에서 렌더링
- [ ] 원본 삭제 후에도 수집 복사본이 영구 생존(스냅샷-삭제-생존 검증)
- [ ] 낚시/수집함이 낚은 사람에게만 보이고 실시간 브로드캐스트 없음
- [ ] 수집함 조회가 본인 스코프로 제한되고 타인 수집함 접근 불가
- [ ] 낚시/수집함 응답에 owner_id 비노출(익명 물고기 신원 유지)
- [ ] 존재하지 않는 물고기 낚기 거부, 중복 낚기 멱등 처리
- [ ] 공유 어항과 분리된 "내 수집함" 화면 + 빈 상태 처리
- [ ] 접근성 보조 수단 제공, 기존 어항 동작 비회귀
- [ ] 모든 인수 시나리오 통과 및 품질 게이트 충족
