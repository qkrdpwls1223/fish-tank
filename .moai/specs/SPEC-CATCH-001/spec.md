# SPEC-CATCH-001: 물고기 낚시 & 개인 수집함 (비파괴 수집)

## HISTORY

- 2026-07-10 (v0.1.0): 최초 작성. SPEC-TANK-001(공유 어항) 위에 얹는 첫 **비공유(private)** 기능. 확정된 요구사항(비파괴 수집, 누구의 물고기든 낚기 가능, 완전 비공개, 별도 "내 수집함" 화면, 스냅샷 영구 보존)을 EARS 형식으로 정리. 핵심 불변식 3개(비파괴 / 프라이버시 / 스냅샷-삭제-생존)를 명시적 요구사항·NFR로 포함. 기존 REQ-OWN-004(소유자 신원 비노출)를 확장 참조.

---

## Goal (목표)

공유 어항(SPEC-TANK-001)에서 헤엄치는 물고기를 **낚아서(catch)** 자신만의 **개인 수집함("내 수집함")** 에 모으는 경험을 제공한다. 낚시는 **비파괴적**이다 — 낚아도 원본 물고기는 공유 어항에 그대로 남으며, 그린 사람은 절대 자신의 물고기를 잃지 않는다. 낚는 행위와 수집함은 **오직 낚은 사람에게만** 보이는, 이 앱 최초의 **비공유 기능**이다.

정서적 핵심(Emotional Core): "내가 그린 물고기는 누가 낚아도 사라지지 않고, 내가 낚은 물고기는 온전히 내 것으로 간직된다."

## Target Audience (대상 사용자)

- 공유 어항을 이미 사용하는 사내 임직원 (SPEC-TANK-001의 인증된 사용자와 동일)
- 사용 맥락: 마음에 드는 물고기(본인 것 또는 타인 것)를 개인적으로 수집·간직, 가벼운 소장 경험

## Scope (범위)

### In Scope (포함)

- 공유 어항의 임의 물고기(본인/타인 무관)를 낚아 개인 수집함에 **복사본(스냅샷)** 추가
- 낚시 시 원본 물고기는 공유 어항에 **그대로 유지**(비파괴)
- 낚는 행위·수집함을 **낚은 사람에게만** 표시(실시간 브로드캐스트 없음)
- 공유 어항과 **분리된** 별도의 "내 수집함" 화면
- 낚은 시점의 그림 데이터/표시 모드/표시 이름을 **독립 스냅샷**으로 영구 저장
- 원본이 나중에 삭제되어도 낚은 복사본은 수집함에 **영구 생존**
- 동일 물고기 중복 낚기에 대한 dedupe 정책

### Out of Scope (제외 — 상세는 아래 Exclusions 참조)

- 낚시 실시간 공유/알림, 수집함 공개/공유, 수집한 물고기 되풀어놓기·거래·랭킹 등

## Functional Requirements (EARS)

> EARS 패턴 표기: **[Ubiquitous]** 항시, **[Event]** 이벤트 기반(WHEN), **[State]** 상태 기반(WHILE), **[Unwanted]** 비정상 처리(IF...THEN), **[Optional]** 선택 기능(WHERE)

### 낚시 / 수집 (CATCH)

- **REQ-CATCH-001 [Event]**: 사용자가 공유 어항의 물고기를 낚을 때(WHEN), 시스템은 그 물고기의 **복사본(스냅샷)** 을 낚은 사용자의 개인 수집함에 추가해야 한다(shall).
- **REQ-CATCH-002 [Ubiquitous]**: 시스템은 사용자가 공유 어항의 **어떤 물고기든**(본인이 그린 것과 타인이 그린 것을 구분 없이) 낚을 수 있게 해야 한다.
- **REQ-CATCH-003 [Ubiquitous] — 비파괴 불변식**: 낚시 동작은 공유 어항의 **원본 물고기를 삭제·수정·숨김·변경하지 않아야 한다**. 낚시로 인해 원본 물고기의 상태나 소유권이 어떤 방식으로도 바뀌어서는 안 된다.
- **REQ-CATCH-004 [Unwanted]**: 사용자가 이미 존재하지 않는(예: 원본이 삭제된) 물고기 ID를 낚으려 하면(IF), 시스템은(THEN) 낚시를 거부하고 "더 이상 존재하지 않음"을 안내하며, 수집함에 항목을 생성하지 않아야 한다.
- **REQ-CATCH-005 [Unwanted] — 중복 낚기(dedupe)**: 사용자가 자신의 수집함에 이미 있는 **동일 원본 물고기**를 다시 낚으려 하면(IF), 시스템은(THEN) 이를 멱등(idempotent)하게 처리하여 중복 항목을 생성하지 않고 "이미 수집함"임을 알려야 한다. (물고기 그림은 편집 불가·불변이므로 재낚기는 동일 사본을 만들 뿐이다 — SPEC-TANK-001 REQ-OWN-005 참조)

### 수집함 조회 (COLL)

- **REQ-COLL-001 [Event]**: 사용자가 "내 수집함"을 열 때(WHEN), 시스템은 그 사용자가 낚은 모든 물고기를 인증된 본인 기준으로 조회하여 표시해야 한다.
- **REQ-COLL-002 [Ubiquitous]**: 시스템은 "내 수집함"을 공유 어항 화면과 **분리된 별도 화면/뷰**로 제공해야 한다.
- **REQ-COLL-003 [Event]**: 수집한 물고기를 표시할 때(WHEN), 시스템은 낚은 시점에 저장된 **스냅샷**(그림 데이터, 표시 모드, 표시 이름)으로 렌더링해야 한다(원본을 재조회하지 않는다).
- **REQ-COLL-004 [State]**: 수집함을 보는 동안(WHILE), 시스템은 각 항목에 대해 낚은 시각(caught_at) 등 수집 메타데이터를 표시해야 한다.
- **REQ-COLL-005 [Unwanted]**: 사용자가 낚은 물고기가 하나도 없으면(IF), 시스템은(THEN) 빈 수집함 상태(안내 문구)를 표시해야 한다.

### 프라이버시 (PRIV) — 비공유 불변식

- **REQ-PRIV-001 [Ubiquitous] — 프라이버시 불변식**: 시스템은 낚는 **행위**와 그 결과인 **수집함**을 오직 낚은 사용자 본인에게만 보이게 해야 한다.
- **REQ-PRIV-002 [Unwanted] — 브로드캐스트 금지**: 시스템은 낚시 이벤트를 실시간 채널(WebSocket)로 **전파하지 않아야 한다**. 낚시는 다른 사용자의 어항 뷰에 어떤 변화도 만들어서는 안 된다(새 이벤트 타입 추가 금지).
- **REQ-PRIV-003 [Ubiquitous]**: 시스템은 수집함 읽기를 인증된 `req.user.userId` 로 엄격히 스코프해야 하며, 사용자는 타인의 수집함을 조회할 수 없어야 한다.
- **REQ-PRIV-004 [Ubiquitous] — 소유자 신원 비노출(REQ-OWN-004 확장)**: 시스템은 낚시·수집함 응답에 원본 물고기(및 어떤 물고기)의 내부 `owner_id` 를 노출하지 않아야 한다. 낚시는 물고기를 누가 그렸는지 드러내서는 안 된다.

### 스냅샷 영속 (SNAP)

- **REQ-SNAP-001 [Event]**: 물고기를 낚을 때(WHEN), 시스템은 그림 데이터와 표시 속성(표시 모드/표시 이름)의 **독립 스냅샷 복사본**을 원본 물고기 레코드와 분리하여 저장해야 한다.
- **REQ-SNAP-002 [Unwanted] — 스냅샷-삭제-생존 불변식**: 원본 물고기가 나중에 공유 어항에서 삭제되면(IF), 시스템은(THEN) 낚은 스냅샷을 낚은 사용자의 수집함에 **영구히 생존**시켜야 한다(간직용 keepsake).
- **REQ-SNAP-003 [Ubiquitous]**: 시스템은 수집 스냅샷이 원본 물고기 레코드의 삭제에 의해 함께 삭제되지 않도록 해야 한다(원본 삭제가 수집 항목으로 cascade 되지 않아야 한다).

## Non-Functional Requirements (NFR)

### 보안 (Security — 인증 기반)

- **NFR-SEC-001 [Ubiquitous]**: 시스템은 낚시·수집함 엔드포인트에 대해 서버 측에서 검증된 신원(`authRequired`, `req.user.userId`)을 사용해야 하며, 낚은 사람(catcher) 신원으로 클라이언트 본문 값을 신뢰해서는 안 된다.
- **NFR-SEC-002 [Ubiquitous]**: 낚시 엔드포인트는 스냅샷 내용을 **서버에 저장된 원본 물고기 레코드**에서 유도해야 하며, 클라이언트가 보낸 그림 데이터로부터 생성해서는 안 된다(임의 콘텐츠를 수집함에 주입하는 것을 방지). 즉 낚시 요청은 원본 물고기 ID만 받는다.
- **NFR-SEC-003 [Unwanted]**: 시스템은 위조/조작된 요청으로 타인의 수집함을 조회·수정하거나, 존재하지 않는/권한 없는 데이터를 수집함에 넣는 행위를 허용하지 않아야 한다.

### 프라이버시 아키텍처 (Privacy Architecture)

- **NFR-PRIV-001 [Ubiquitous]**: 낚시/수집함은 이 앱 최초의 비공유 기능이므로, 시스템은 공유 경로(브로드캐스터, `GET /api/fish` 스냅샷)와 **분리된** 사용자 스코프 읽기 경로만 사용해야 한다. 수집 데이터가 공유 어항의 실시간/스냅샷 응답에 절대 섞이지 않아야 한다.

### 비회귀 / 공존 (Non-Regression & Coexistence)

- **NFR-COMPAT-001 [Ubiquitous]**: 시스템은 SPEC-TANK-001의 기존 공유 어항 동작(추가/삭제/먹이주기/실시간 전파, `toPublicFish`/`toViewerFish` 투영, 소유권 삭제 규칙)을 변경·저하시키지 않아야 한다. 낚시 기능은 기존 동작에 대해 비회귀여야 한다.

### 렌더링 성능 (Rendering Performance)

- **NFR-PERF-001 [State]**: 수집함에 다수의 물고기가 존재하는 동안(WHILE), 시스템은 수집함 화면의 렌더링 성능 저하를 방지해야 한다(SPEC-TANK-001 NFR-PERF-001과 동일한 렌더링 상한/최적화 원칙 준용).

### 접근성 (Accessibility)

- **NFR-A11Y-001 [Ubiquitous]**: 시스템은 낚시 동작과 "내 수집함" 화면에 대해 접근성 대체 수단을 제공해야 한다(키보드 조작 경로, 명확한 라벨/안내 문구, 충분한 색 대비 — SPEC-TANK-001 NFR-A11Y-001 준용).

## Exclusions (What NOT to Build)

초기 범위에서 명시적으로 제외한다. (범위 크립 방지)

- **낚시 실시간 공유/알림**: 낚시 이벤트는 절대 브로드캐스트하지 않는다 (REQ-PRIV-002). 다른 사용자에게 "누가 무엇을 낚았다"를 알리는 기능은 만들지 않는다.
- **수집함 공개/공유**: 타인의 수집함 열람, 수집함 공유 링크, 랭킹/리더보드는 만들지 않는다 (REQ-PRIV-001/003).
- **원본 물고기 제거형 낚시(파괴적 수집)**: 낚아서 원본을 가져가거나 어항에서 없애는 메커니즘은 만들지 않는다 — 낚시는 항상 비파괴다 (REQ-CATCH-003).
- **수집 물고기 편집/되풀어놓기/거래**: 낚은 스냅샷을 다시 그리거나, 공유 어항으로 되돌리거나, 사용자 간 교환하는 기능은 만들지 않는다.
- **원본 소유자 신원 공개**: 낚시를 통해 물고기를 누가 그렸는지 조회/표시하는 기능은 만들지 않는다 (REQ-PRIV-004, SPEC-TANK-001 REQ-OWN-004).
- **수집 개수 제한/포인트/게임화**: 수집 상한, 획득 포인트, 도감 진행도 등 게임 메커니즘은 초기 범위 밖이다.

## Assumptions (가정)

- 낚시·수집함은 SPEC-TANK-001과 동일한 인증 컨텍스트(`authRequired` → `req.user.userId` = 토큰 `oid`)를 재사용한다.
- 원본 물고기 그림 데이터는 이미 서버에서 검증되어 저장된 상태이므로, 스냅샷은 검증 완료된 데이터의 서버 측 복사다(재검증 불필요, NFR-SEC-002).
- 저장은 기존과 동일한 PostgreSQL(`pg` Pool)을 사용하며, 새 테이블(예: `caught_fish`)을 추가한다.
- 수집함 렌더링은 기존 캔버스 렌더링 자산/원칙을 재사용할 수 있다.

## Dependencies (의존성)

- SPEC-TANK-001 (공유 어항) — 인증(`server/src/auth/authMiddleware.js`), 물고기 저장소(`fishRepository.js` / `pgFishRepository.js`), 공개 투영(`server/src/fish/publicFish.js`), 라우트(`server/src/routes/fish.js`), 마이그레이션(`server/migrations/001_create_fish.sql`).
- PostgreSQL (`pg` Pool) — 새 `caught_fish` 테이블 및 병렬 저장소 페어.
- 프론트엔드: React + Vite, 캔버스 렌더링(`client/src/tank/`, `client/src/fish/fishApi.js`, `client/src/App.jsx`).

## Suggested Implementation Surface (제안 — 최종 형태는 Run 단계에서 확정)

> 이 절은 구현 힌트이며 계약이 아니다. 구체 스키마/함수/API 형식은 Run 단계에서 결정한다.

- **새 테이블 `caught_fish`**: catch id(UUID PK), `catcher_id`(토큰 oid, TEXT), 낚은 시점 스냅샷(`drawing` JSONB, `display_mode`, `display_name`), `source_fish_id`(dedupe용, 원본 삭제와 무관하게 유지), `caught_at`. 원본 `fish` 행 생명주기와 독립(FK cascade 없음, REQ-SNAP-003).
- **새 엔드포인트(예)**: `POST /api/fish/:id/catch`(원본 ID만 수신, 서버가 스냅샷 생성), `GET /api/me/catches`(본인 스코프 조회). 모두 `authRequired`.
- **새 저장소 페어(예)**: `InMemoryCatchRepository`(테스트) + `PgCatchRepository`(프로덕션) — create/listByCatcher/existsForCatcher 등.
- **새 프론트엔드 화면**: "내 수집함" 뷰(`client/src/App.jsx` 상위 뷰 전환), 어항에서의 낚시 트리거 UI.
