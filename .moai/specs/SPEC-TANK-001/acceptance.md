# 인수 기준: SPEC-TANK-001 공유 어항

> Given-When-Then 형식. 각 시나리오는 관찰 가능한 증거를 기준으로 검증한다.

## Scenario 1: Teams SSO 로그인 및 앱 진입 (REQ-AUTH-001, REQ-AUTH-002, REQ-RT-004)

- **Given** 사용자가 사내 Teams에 로그인된 상태이고 어항 탭을 연다
- **When** 앱이 로드된다
- **Then** Teams JS SDK SSO로 세션과 사용자 신원이 획득되고, 저장된 모든 물고기가 어항에 렌더링된다

## Scenario 2: 이름 표시 물고기 생성 및 실시간 반영 (REQ-DRAW-001~003, REQ-AUTH-003, REQ-PERSIST-001, REQ-RT-001)

- **Given** 로그인 사용자 A와 다른 접속 사용자 B가 동시에 어항을 보고 있다
- **When** A가 캔버스에 물고기를 그리고 "이름 표시"로 등록한다
- **Then** 그림이 DB에 영구 저장되고, A와 B 모두의 어항에 해당 물고기가 수동 새로고침 없이 즉시 나타나 헤엄친다

## Scenario 3: 익명 물고기 생성 및 소유권 내부 추적 (REQ-AUTH-003, REQ-OWN-001, REQ-OWN-004)

- **Given** 로그인 사용자 A가 어항을 보고 있다
- **When** A가 물고기를 그리고 "익명"으로 등록한다
- **Then** 어항과 정보 표시 어디에도 A의 이름/신원이 노출되지 않고, 내부적으로는 A의 Teams 사용자 ID가 소유자로 저장된다

## Scenario 4: 본인 익명 물고기 삭제 (REQ-OWN-002, REQ-RT-002)

- **Given** 사용자 A가 이전에 익명으로 등록한 물고기가 어항에 있다
- **When** A가 로그인 상태에서 해당 익명 물고기의 삭제를 요청한다
- **Then** 물고기가 삭제되고, 접속 중인 모든 사용자의 어항에서 실시간으로 사라진다

## Scenario 5: 타인 물고기 삭제 거부 (REQ-OWN-003, NFR-SEC-002)

- **Given** 사용자 B가 생성한 물고기(이름/익명 무관)가 어항에 있다
- **When** 사용자 A가 해당 물고기의 삭제를 시도한다
- **Then** 시스템은 삭제를 거부하고 권한 없음을 안내하며, 물고기는 그대로 유지된다

## Scenario 6: 먹이주기 및 물고기 정보 조회 (REQ-INT-001, REQ-INT-002)

- **Given** 어항에 이름 물고기와 익명 물고기가 함께 있다
- **When** 사용자가 먹이주기를 실행하고 각 물고기를 클릭/호버한다
- **Then** 물고기가 먹이 반응 애니메이션을 보이고, 이름 물고기는 표시 이름과 생성 정보가, 익명 물고기는 "익명"으로 정보가 표시된다

---

## Edge Cases (엣지 케이스)

- **익명 물고기 소유권/삭제**: 익명 물고기도 내부 소유자 ID로 삭제 권한을 검증한다. 소유자 본인만 삭제 가능하며, 타인은 삭제 불가하고 신원은 절대 노출되지 않는다. (REQ-OWN-001~004)
- **빈/무효 그림**: 획이 없거나 최소 크기 미달인 그림은 등록이 거부되고 사유가 안내된다. (REQ-DRAW-004)
- **다수 물고기 렌더링**: 물고기가 수백 마리 이상일 때도 렌더링 성능이 유지되어야 한다(프레임 저하/멈춤 없음). (NFR-PERF-001)
- **삭제 실시간 동기화**: 한 사용자가 삭제하면 모든 접속 사용자 화면에서 즉시 제거된다. (REQ-RT-002)
- **Teams SSO 실패**: 토큰 획득/검증 실패 시 오류 상태·재시도 경로가 표시되고 생성/삭제/먹이주기가 비활성화된다. (REQ-AUTH-004)
- **실시간 연결 끊김/재연결**: 연결이 끊긴 뒤 재연결되면 현재 어항 상태와 자동 재동기화된다(수동 새로고침 불필요). (REQ-RT-003)
- **위조 삭제 요청**: 클라이언트에서 조작된 소유자 정보로 삭제를 시도해도 서버가 실제 인증 신원으로 소유권을 검증해 차단한다. (NFR-SEC-001, NFR-SEC-002)
- **대용량/악성 그림 데이터**: 크기 상한 초과 또는 검증 실패 시 저장이 거부된다. (NFR-SEC-003)

## Quality Gate Criteria (품질 게이트)

- 모든 EARS 기능 요구사항(REQ-*)에 대응하는 인수 시나리오 또는 엣지 케이스가 존재한다.
- 테스트 커버리지 목표: 85% 이상 (TDD, `quality.yaml` 기준).
- LSP 오류/타입 오류/린트 오류 0 (Run 단계 기준).
- 서버 측 인증·소유권 검증이 모든 쓰기 경로에 적용됨을 테스트로 증명한다(NFR-SEC-001/002).
- 실시간 전파 지연 목표 p95 ≤ 1초 검증(NFR-RT-001).

## Definition of Done (완료 정의)

- [ ] Teams 탭에서 SSO 로그인 후 어항 진입 및 전체 물고기 렌더링 동작
- [ ] 자유 드로잉으로 이름/익명 물고기 생성 및 DB 영구 저장
- [ ] 물고기 추가/삭제가 모든 접속 사용자에게 실시간 반영
- [ ] 본인 물고기(익명 포함) 삭제 가능, 타인 물고기 삭제 거부
- [ ] 익명 물고기 신원 비노출 + 내부 소유권으로 삭제 검증
- [ ] 먹이주기 및 물고기 정보 조회 동작
- [ ] 빈/무효 그림, SSO 실패, 재연결, 위조 삭제 등 엣지 케이스 처리
- [ ] 접근성 보조 수단 제공(NFR-A11Y-001)
- [ ] 모든 인수 시나리오 통과 및 품질 게이트 충족
